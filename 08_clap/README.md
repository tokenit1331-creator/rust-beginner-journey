**Краткая выжимка (топ-5):**

| Совет | Почему это важно |
|---|---|
| 🪄 **Всегда используй Derive** | Меньше кода, ошибки на этапе компиляции, читаемость |
| ✅ **unwrap() в main() — норма** | clap сам вызывает exit(2) при ошибке |
| 📋 **ValueEnum вместо String** | Компилятор отлавливает опечатки, автодополнение в help |
| 🔐 **env для API-ключей** | Ключи не светятся в истории bash и ps aux |
| 🔢 **Count для verbose** | -v, -vv, -vvv одной строкой кода |

## ❓ Хватит ли этого материала чтобы изучить clap?

**Да, на 90% для повседневной работы — более чем достаточно.**

Этот урок покрывает всё, что нужно **новичку и middle-разработчику**:

| Что ты умеешь после урока | Уровень |
|---|---|
| Позиционные аргументы, опции, флаги | 🟢 Базовый |
| ValueEnum (строгие списки) | 🟢 Средний |
| Субкоманды (как в git, cargo, docker) | 🟡 Средний |
| Валидация: range, кастомные парсеры | 🟡 Средний |
| conflicts_with / requires | 🟠 Продвинутый |
| env, Count, hide, help_template | 🟠 Продвинутый |
| Shell completion | 🔴 Экспертный |

**Что осталось за кадром** (менее 10%):
- ArgGroup (сложные группы аргументов)
- flatten (переиспользование структур)
- Вложенные субкоманды (sub-subcommands)
- Трейт `FromArgMatches` для экстремальной кастомизации

> Но для 95% реальных проектов этого хватит с головой. Остальное — гуглится по мере необходимости.

## 📊 Сравнение: Derive vs Builder

| Критерий | Derive (мы используем) | Builder |
|----------|----------------------|---------|
| Код | 🟢 Минимум | 🔴 Много boilerplate |
| Читаемость | 🟢 Как структура данных | 🟡 Как цепочка вызовов |
| Ошибки компиляции | 🟢 clap проверяет типы | 🟡 Ошибки только в рантайме |
| Гибкость | 🟡 Достаточно для 95% задач | 🟢 Полный контроль |
| Для новичка | ✅ **Выбираем его** | ⚠️ Сложнее |

> **Почему мы выбрали Derive:**
> 1. Меньше кода = меньше ошибок
> 2. Компилятор проверяет типы аргументов на этапе сборки
> 3. Легче читать: вся структура CLI видна в одном месте
> 4. 95% реальных проектов используют Derive
