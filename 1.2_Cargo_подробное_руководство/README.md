# 📦 Cargo — система сборки и зависимостей Rust (подробное руководство)

Cargo — это не просто «менеджер пакетов», а полноценная система, которая управляет всем жизненным циклом Rust-проекта: создание, сборка, запуск, тестирование, добавление зависимостей, публикация. Для системных проектов (драйвера, ядра, компиляторы, утилиты) Cargo является стандартом де-факто. Без него собирать проекты с зависимостями было бы крайне сложно.

---

## 1. Создание проекта: `cargo new`

Создаёт новый бинарный (по умолчанию) или библиотечный проект с типовой структурой.

### Команды:

```bash
cargo new my_project          # создаёт бинарный проект (src/main.rs)
cargo new --lib my_lib        # создаёт библиотечный проект (src/lib.rs)
cargo new --bin my_tool       # явно указать бинарный (то же, что по умолчанию)
```

Сгенерированная структура:

```
my_project/
├── Cargo.toml      # метаданные проекта, зависимости
├── src/
│   └── main.rs     # точка входа (fn main)
└── .gitignore      # игнорирование target/ и прочего
```

**`Cargo.toml`** — главный манифест. Пример минимального:

```toml
[package]
name = "my_project"
version = "0.1.0"
edition = "2021"

[dependencies]
```

---

## 2. Компиляция: `cargo build`

Собирает проект и все зависимости.

- **`cargo build`** → отладочная сборка (без оптимизаций, с отладочными символами). Результат: `target/debug/my_project`.
- **`cargo build --release`** → оптимизированная сборка (флаги `-O`, без отладочных символов). Результат: `target/release/my_project`.
- **`cargo build --target <triple>`** → кросс‑компиляция (например, `--target wasm32-unknown-unknown`).

---

## 3. Запуск: `cargo run`

Собирает (при необходимости) и сразу запускает исполняемый файл.

```bash
cargo run            # запуск в режиме debug
cargo run --release  # запуск оптимизированной версии
```

Если программе нужны аргументы командной строки, их пишут после `--`:

```bash
cargo run -- --help      # передать аргумент --help самой программе
```

---

## 4. Проверка без генерации кода: `cargo check`

Быстро проверяет код на ошибки компиляции, **не** создавая исполняемый файл. Это значительно экономит время, особенно в больших проектах.

```bash
cargo check
cargo check --release
```

---

## 5. Добавление зависимостей: `cargo add`

Добавляет крейт из [crates.io](https://crates.io) в `[dependencies]`. Команда встроена в Cargo начиная с версии 1.62 (ранее требовался отдельный `cargo-edit`).

```bash
cargo add serde                # добавить последнюю версию
cargo add serde --vers 1.0.0   # конкретная версия
cargo add serde --features derive   # включить фичу "derive"
cargo add --dev anyhow         # добавить в dev-зависимости (только для тестов)
cargo add --build cc           # добавить в build-dependencies (для сборочных скриптов)
```

После выполнения команды в `Cargo.toml` появляется строчка:

```toml
[dependencies]
serde = { version = "1.0.217", features = ["derive"] }
```

**Альтернатива** — править `Cargo.toml` вручную, но `cargo add` удобен и минимизирует ошибки.

---

## 6. Тестирование: `cargo test`

Запускает все тесты в проекте (как интеграционные, так и модульные).

```bash
cargo test            # все тесты
cargo test -- --nocapture   # показывает вывод println! в тестах
cargo test test_name        # запустить только тесты с указанной строкой в имени
```

## 7. Структура `Cargo.toml` – подробно

### 7.1 Секция `[package]`

Обязательные поля:

- **`name`** – имя проекта (должно быть валидным идентификатором Rust).
- **`version`** – семантическая версия (например, `0.1.0`).
- **`edition`** – выпуск Rust (`2015`, `2018`, `2021`, `2024`). Определяет набор синтаксических правил.

Опциональные, но полезные поля:

- **`authors`** – список авторов.
- **`description`** – описание проекта (отображается на crates.io).
- **`license`** – например, `MIT` или `Apache-2.0`.
- **`repository`** – URL репозитория.
- **`readme`** – путь к README файлу.
- **`keywords`** – массив строк для поиска на crates.io.
- **`categories`** – категории проекта.

### 7.2 Секции зависимостей

- **`[dependencies]`** – основные зависимости, нужные и для бинарного файла, и для библиотеки.
- **`[dev-dependencies]`** – зависимости только для тестов, примеров и бенчмарков.
- **`[build-dependencies]`** – зависимости, используемые в `build.rs` (скрипт сборки).

Формат записи зависимостей:

```toml
[dependencies]
simple = "0.11"          # версия совместимая с 0.11.x (не 0.12)
exact = "=1.2.3"         # точная версия
minimal = ">=1.0, <2.0"  # диапазон
git = { git = "https://github.com/user/repo", branch = "main" } # из git
local = { path = "../my_local_crate" }   # локальный путь
```

### 7.3 Профили сборки (`[profile.*]`)

Профили определяют параметры компиляции для разных режимов.

| Профиль | Назначение |
|---------|-------------|
| `dev`   | `cargo build` по умолчанию (быстрая компиляция, мало оптимизаций) |
| `release` | `cargo build --release` (высокие оптимизации, долго компилируется) |
| `test`   | `cargo test` (аналогичен `dev`, но для тестов) |
| `bench`  | `cargo bench` |

Пример переопределения параметров в `Cargo.toml`:

```toml
[profile.dev]
opt-level = 0          # уровень оптимизации (0-3)
debug = true           # отладочные символы
lto = false            # link-time optimization
panic = "unwind"       # механизм паники (unwind или abort)

[profile.release]
opt-level = 3
debug = false
lto = "thin"           # тонкое LTO
panic = "abort"        # лучше для небольших бинарных файлов (ядро, драйверы)
```

Для системного программирования (ядро, драйверы) часто используют `panic = "abort"` в `release`, чтобы уменьшить размер бинарного файла и избежать возни с раскруткой стека.

### 7.4 Фичи (features)

Фичи – это условная компиляция: можно включать/отключать части кода зависимостей или собственного крейта.

```toml
[features]
default = ["std"]      # фичи, включаемые по умолчанию
std = []               # фича без зависимостей
alloc = ["core2/alloc"]  # фича, которая активирует alloc у крейта core2
```

Использование в коде:

```rust
#[cfg(feature = "std")]
extern crate std;

#[cfg(not(feature = "std"))]
#[panic_handler]
fn panic(_info: &core::panic::PanicInfo) -> ! { loop {} }
```

В командной строке:

```bash
cargo build --features "std, alloc"
cargo build --no-default-features
```

Для системного программирования фичи позволяют отключать стандартную библиотеку (`no_std`) и подключать только `core` или `alloc`.

## 8. Продвинутые темы

### 8.1 `build.rs` – скрипт сборки

Если проекту нужна сложная логика при компиляции (генерация кода, проверка окружения, компиляция C/C++ библиотек), создаётся файл `build.rs` в корне проекта. Cargo скомпилирует и запустит его перед сборкой основного крейта.

```rust
// build.rs
fn main() {
    println!("cargo:rerun-if-changed=src/c_code.c");
    cc::Build::new().file("src/c_code.c").compile("myc");
}
```

### 8.2 Рабочее пространство (workspace)

Если вы разрабатываете несколько взаимосвязанных крейтов (например, для ядра: `hal`, `alloc`, `core_ext`), создайте `Cargo.toml` на верхнем уровне с секцией `[workspace]`:

```toml
[workspace]
members = [
    "crates/hal",
    "crates/alloc",
    "kernel",
]
```

В подкаталогах должны быть свои `Cargo.toml`. Команды `cargo build` из корня рабочей области соберут всё сразу.

### 8.3 Условная компиляция на основе `target_os`, `target_arch`

Без фичей можно использовать встроенные атрибуты:

```rust
#[cfg(target_os = "linux")]
fn platform_specific() { ... }

#[cfg(target_arch = "x86_64")]
#[inline(never)]
fn arch_specific() { ... }
```

## 9. Пример системного проекта: `no_std` драйвер

Создадим проект, который будет собран под `thumbv7m-none-eabi` (микроконтроллер ARM) и не зависит от стандартной библиотеки. Нам понадобится `nightly` и настройки в `Cargo.toml`.

**`Cargo.toml`:**

```toml
[package]
name = "cortex_m_driver"
version = "0.1.0"
edition = "2021"

[dependencies]
cortex-m = "0.7"          # поддержка Cortex-M
cortex-m-rt = "0.7"       # минимальный рантайм

[profile.release]
panic = "abort"
lto = true
opt-level = "z"           # минимизация размера
```

Компиляция:

```bash
rustup target add thumbv7m-none-eabi
cargo build --release --target thumbv7m-none-eabi
```

## 10. Часто задаваемые вопросы

**Q: В чём разница между `cargo check` и `cargo build`?**  
A: `check` проверяет типы и синтаксис, но не генерирует машинный код, поэтому работает быстрее.

**Q: Как понизить версию зависимости?**  
A: Укажите точную версию в `Cargo.toml` или выполните `cargo update -p имя_крейта --precise <версия>`.

**Q: Что означает ~ и ^ в версиях?**  
- `^1.2.3` — совместимая с 1.2.3, разрешает обновления до версии <2.0.0.  
- `~1.2.3` — только патч-релизы (до 1.3.0).  
- `1.2.3` — то же, что `^1.2.3`.

**Q: Как собрать статическую библиотеку (`.a`) вместо исполняемого файла?**  
Добавьте в `Cargo.toml`:

```toml
[lib]
crate-type = ["staticlib", "rlib"]
```

**Q: Можно ли использовать Cargo в проектах, не публикуя код на crates.io?**  
Да, зависимости могут быть локальными (`path = ..`) или из git-репозиториев.

## 11. Заключение

Cargo — основа любого Rust-проекта. Она позволяет:

- Создавать проекты за секунду.
- Управлять сложными графами зависимостей.
- Настраивать профили сборки для разных целей (размер, скорость, отладка).
- Использовать условную компиляцию через фичи.
- Строить проекты без стандартной библиотеки (например, ядро ОС).

Понимание `Cargo.toml`, профилей и фич критически важно для системного программирования. Без Cargo современная Rust-экосистема была бы невозможна. Удачи в ваших проектах! 🚀
